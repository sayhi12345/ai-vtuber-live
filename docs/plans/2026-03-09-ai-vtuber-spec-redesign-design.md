# AI-VTuber SPEC 重構設計

日期：2026-03-09

## 目標

將現有 `SPEC.md` 從偏技術方案的文件，重構為單一份可同時服務產品、設計與工程協作的雙層文件：

1. 前半部是產品需求層，先定義目標用戶、核心場景、MVP、成功指標與驗收標準。
2. 後半部是技術方案層，說明系統架構、模組分工、延遲預算與選型原則。

## 採用方案

採用 `Dual-layer single doc`：

- 同一份文件完成需求對齊與技術落地。
- 產品決策與技術決策分章節管理，避免混在一起。
- 技術內容從「模型/供應商清單堆疊」改為「能力需求 + 選型原則 + 基準方案」。

## 新結構

1. 執行摘要
2. 產品背景與願景
3. 目標用戶與核心場景
4. 問題定義與產品目標
5. MVP 範圍、非目標與版本策略
6. 功能需求與驗收標準
7. 成功指標、風險與里程碑
8. 技術方案總覽
9. 系統架構與模組規格
10. 延遲預算、可靠性與安全
11. 技術選型原則與建議基線
12. 開放問題與附錄

## 重構原則

- 先講「為什麼做」，再講「怎麼做」。
- 所有 MVP 需求都要可驗收，不寫空泛能力描述。
- 對時間敏感的模型跑分、供應商排名不作為主體內容，只保留可替換的選型框架。
- 保留原文件對低延遲、串流回應、表情同步與內容安全的核心重點。

## 預期輸出

- 一份更像 PRD + SPEC 的 `SPEC.md`
- 可直接用於需求評審、工程拆解與後續實作計畫

## 範圍修正

在同日澄清後，第一版產品範圍進一步收斂為：

- 不接直播平台
- 以文字輸入為主
- 提供對話式 Web App
- 以角色體驗為優先，而非直播互動吞吐
- 保留未來嵌入 OBS 的能力，但不作為 MVP 主流程

因此後續 `SPEC.md` 需以 `character-first` 的互動產品為主體，而不是直播中台或聊天室接入系統。
